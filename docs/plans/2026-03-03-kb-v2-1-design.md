# 企业级AI知识库系统 v2.1 设计修订稿

> 版本：v2.1 | 日期：2026-03-03 | 基于 `technical-proposal.md` 的在线编辑与出网策略修订

## 1. 本次修订决策（已确认）

- **出网策略调整为 A**：AI 问答与索引阶段允许发送“最小必要、脱敏后的文本片段”和问题上下文，不再要求“文件完整内容绝对不出网”。
- **文件编辑能力**：新增 `DOC/DOCX/XLS/XLSX/PPT/PPTX` 的在线编辑能力，采用 **ONLYOFFICE**。
- **编辑模式**：先实现**单人编辑**，不做实时协同；仅用于 v2.1 首期。
- **保存行为**：文档保存成功后立即触发重索引（`parse_status=processing` -> 重解析 -> 重分块 -> 重向量化）。
- **编辑权限**：
  - 文件所有者始终可编辑
  - 超级管理员可编辑全局全部文件
  - 部门主管可编辑本部门文件
  - 被共享文件在分享时默认可编辑
- **共享机制默认值**：共享文件默认开放“可编辑”，并在记录里保留 `can_edit=true`。

## 2. 架构与服务边界

### 2.1 服务扩展

- 现有服务基线从 `6` 个扩展到 `7` 个，新增 `onlyoffice`。
- 交互链路：
  - `前端（文件页） -> backend(签发编辑会话) -> ONLYOFFICE editor -> callback -> backend(保存回写) -> MinIO + Postgres`。
- 旧约束 `doc_chunks` 和 `index_tree` 的一致性保持不变，但新增“编辑保存触发重索引”规则。

### 2.2 设计原则变化

- 编辑保存必须先经过服务端权限二次校验，不依赖前端 UI 控制作为唯一边界。
- 所有编辑后的新版本必须写 `file_versions`，并更新 `files.current_version`。
- 对同一次编辑会话，仅允许保存到“当前会话来源版本”，防止并发旧版本覆盖。

## 3. 数据模型变更（v2.1）

### 3.1 共享与权限

- `folder_permissions` 中新增可选编辑标记建议（最小变更）：`allow_edit BOOLEAN DEFAULT true`。
- 文件级共享默认可编辑，无需显式传 `can_edit=false`。

### 3.2 编辑会话（建议新增）

- 新增表 `doc_edit_session`（可选）：用于对齐回调和版本并发，字段建议：
  - `id`、`file_id`、`user_id`
  - `file_version_id`、`edit_token`
  - `status(active/closed/expired)`、`expires_at`
  - `created_at/updated_at`
- 若不引入该表，至少要在缓存层记录 `edit_token -> file_id -> user_id -> version` 的映射并可回收。

## 4. API与交互流程改造（关键）

### 4.1 新增流程

1. `POST /api/v1/files/{file_id}/edit/start`
   - 校验登录、权限、文件类型、文件状态
   - 生成编辑会话（JWT/随机 token）与 `file_version_id`
   - 返回 ONLYOFFICE 初始化参数（下载链接、回调地址、权限）
2. 前端加载编辑器（iframe）进行编辑
3. 编辑器回调 `POST /api/v1/files/{file_id}/edit/callback`
   - 校验签名/来源+token+版本
   - `status`=2/6 时下载新文件、写入 MinIO 新版本对象
   - 更新 `files.current_version`
   - 触发后台异步重解析任务
4. 回调成功返回 `{error:0}`，失败返回 `{error:1}` 并带错误码

### 4.2 文件状态联动

- 编辑开始：`files.status` 标记为 `processing`（可选，避免误触发问答）
- 保存回写成功：新版本入 `file_versions`，旧版本保留；同时更新 `doc_chunks` 与 `index_tree`。
- 重索引任务失败：`parse_status=failed`，记录 `parse_error`，允许用户人工重试。

## 5. 处理链条中的一致性与安全

- 回调保存是系统事实来源（single source of truth）；前端状态只作展示，不能直接决定持久化。
- 任何 `status=2/6/7` 保存动作必须执行：
  - 重新校验用户权限（owner/主管/管理员/共享编辑者）
  - 二次 MIME/Magic Bytes 校验
  - 版本快照比对，避免旧会话回写新位置。
- 回调接口记录审计：`doc_edit_start`、`doc_edit_save`、`doc_edit_save_failed`。

## 6. 资源与性能

- 服务规模由 6 增到 7 个容器，内存预算从原估算上调。
- 推荐内存预估（30人）：
  - PostgreSQL: `512MB-1GB`
  - FastAPI: `256MB`
  - Redis: `512MB`
  - MinIO: `256MB`
  - Nginx + 前端: `128MB`
  - ONLYOFFICE: `800MB-1.2GB`
  - 合计：`2.4-3.2GB`（保留余量）

## 7. 测试策略（变更优先）

- **单点能力测试**
  - 编辑权限：仅允许 owner/主管/管理员/共享人打开编辑态。
  - 回调幂等：重复 `status=2` 不产生重复创建版本。
  - 旧版本回写：会话版本不一致时拒绝。
- **组合测试**
  - 上传->编辑->保存->重索引->问答，检索不返回旧内容。
  - 并发编辑防护：同一文件快速两次编辑会话，后进入的会话在回调时被拒绝。
- **手工验证**
  - 在线打开/保存可见编辑后的文件，并在文件树与版本列表中看到新版本。
  - 查看编辑失败告警及重试入口。
